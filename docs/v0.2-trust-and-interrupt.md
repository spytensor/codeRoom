# CodeRoom v0.2 — Trust, Interrupt, Parallel

Status: design doc, **anchor for v0.2 PRs**. Lock spec here before code lands.
Anything that contradicts this doc goes to `docs/proposed-amendments.md` first.

Companion to `docs/architecture.md` — that one is v0.1 constitution; this one
records the v0.2 amendment that grew out of the May 2026 codex-event-routing
fix (PR #80) and the design conversation that followed it.

## TL;DR

v0.2 commits to three coupled changes that share one root: drain the role
turn out of the REPL's main control flow.

1. **Trust** — delete `PER_TURN_TIMEOUT`. The wrapper does not adjudicate when
   a role has "taken too long". Modern Claude / Codex / Gemini models
   self-terminate; the wrapper is not allowed to override that.
2. **Interrupt** — every running turn is cancellable from the user prompt:
   `/stop @role`, `/stop` (all in-flight), and `Ctrl-C` (all in-flight, then
   the second press exits the REPL).
3. **Parallel** — multiple roles can have active turns concurrently. Role
   internally serial (one engine session, one turn at a time); roles
   externally parallel (no global drain).

These three are inseparable. Ship one without the others and the system
regresses: trust without interrupt = users locked into runaway turns; trust
without parallel = REPL prompt blocked by long turns; parallel without
interrupt = no escape hatch when a role goes off the rails.

## Why now (and why all three)

The v0.1.18 → PR #80 path made codex tool events visible in real time. Once
work is visible, two things follow immediately:

- The 5-minute `PER_TURN_TIMEOUT` becomes user-hostile. A real security scan
  or refactor takes 10–30 minutes. The user can see codex working — the
  wrapper killing it on a wall clock is paternalistic and violates the
  Role Invariance Principle's promise that **the user is the routing source**.
- The user wants to react to what they see. "I noticed @security probing
  the wrong subsystem at minute 4" is only useful if the user can stop it.
  Without interrupt support, observation is decorative.
- Interrupting a single role only works if the REPL prompt is reachable
  while that role runs. Today the prompt is blocked behind `drain_one_turn`.

So the three changes share the same prerequisite (drain comes out of the
main loop), and they share the same justification (user-as-driver).

## Trust: what it covers, what it doesn't

"Completely trust the model" — the working stance — applies to **model
behavior**. It does **not** apply to the IPC / RPC plumbing. Three layers,
treated separately:

| Layer | Trusted? | What v0.2 does |
| --- | --- | --- |
| Model self-termination (claude / codex / gemini) | Yes | No timeout in REPL or adapter for "model took too long" |
| Engine MCP / stdio protocol (codex `mcp-server`, gemini `--prompt`, …) | No | Adapter keeps an **idle** timeout as a wedge detector (PR #80 already does this for codex) |
| User control | n/a | `/stop`, Ctrl-C, `/refresh`, `/exit` are all reachable while a turn runs |

The adapter idle timeout is **not** a wall clock; it resets on every event
flowing for the in-flight request. It only fires on genuine stdio silence
(no events for N minutes), which empirically only happens when the engine
process or the bridge is wedged. Reframe (in code comments and doc) from
"safety net for runaway models" to "stdio protocol watchdog".

## Architecture changes

### A. Turn IDs

CREP events today carry only `role`. v0.2 adds `turn_id: String` (a ULID
or short uuid; renderer-opaque) to every CREP event tied to a specific
turn:

- `WorkTitle`, `RoleSpoke`, `ToolCallProposed`, `ToolCallExecuted`,
  `PermissionDenied` — gain `turn_id`.
- `RoleStarted`, `RoleStopped` — unchanged (lifecycle, not per-turn).

Why: in the parallel model a single role can have queued turns, and the
auto-router can spawn turns concurrent with user-driven ones. Without a
turn_id, the renderer cannot route events to the right WorkCard and the
log is ambiguous.

This is a CREP **breaking change**. v0.x is the iteration zone; the on-disk
`messages.jsonl` shape gets a new field. Old logs replay fine (missing
field tolerated; collapsed to a single synthetic turn per role).

### B. Interrupt signal channel

`RoleHandle` today exposes `stop_tx: oneshot<StopReason>`. One-shot, one
meaning: kill the role process. v0.2 splits the semantics:

- `stop_tx: oneshot<StopReason>` — unchanged. **Kills the role.** Used by
  `/refresh`, `/exit`, REPL teardown.
- `interrupt_tx: mpsc::Sender<TurnId>` — new. **Cancels the named turn,
  role process stays alive.** Used by `/stop`, `/stop @role`, Ctrl-C.

Adapter contract on receiving `InterruptTurn(turn_id)`:

- If `turn_id` is the in-flight turn: cancel the engine request (codex →
  send JSON-RPC `notifications/cancelled` for the request id; cc → write
  a stream-json `interrupt` message; gemini → kill the per-turn child).
- If `turn_id` is queued but not started: drop it from the queue, emit
  `RoleSpoke { turn_id, text: "[interrupted before start]", … }` followed
  by `WorkCard::Interrupted` for the renderer.
- Either way, the role stays usable for the next `/dispatch` from the
  REPL.

Concrete adapter cancel paths are listed in §F.

### C. Role-internal serial, role-external parallel

Each role process owns **one** engine session at a time. So per-role
turns must be serial (cc and codex don't support concurrent in-session
prompts). Across roles there is no coordinator — each role drains its own
turn queue independently, on its own task.

Implementation shape:

```text
RunningRole {
    name: String
    handle: RoleHandle
    pending: VecDeque<DispatchedTurn>   // queued + in-flight
    drain_task: JoinHandle<()>          // pulls pending, dispatches, awaits completion
}
```

The REPL no longer drains turns directly. It calls `dispatch(role, prompt)`
which appends to `pending` and (if the drain task is idle) wakes it. The
drain task observes adapter events and folds them into the per-turn
WorkCard state shared with the renderer via `Arc<Mutex<…>>` keyed by
`turn_id`.

### D. REPL main loop

Goes from "dispatch + drain in one select!" to "dispatch only, render off
the bus". Sketch:

```text
loop {
    select! {
        ctrl_c => interrupt_all_in_flight(); maybe_exit_on_double_press()
        line = stdin => parse_command_or_dispatch()
        evt = bus => render(evt); update_status_region()
        winch = signal::winch => repaint()
    }
}
```

The REPL is now **always interactive**. Even with eight turns in flight,
the prompt accepts new commands. No `tokio::time::timeout` anywhere in
this loop.

### E. Render strategy: append-only, multi-role status

Reaffirms what the gutter work (#0afa871) already enables. v0.2 commits
to it explicitly:

- **Per-event lines append in arrival order, prefixed by the speaking
  role's color gutter.** Multi-role event streams interleave; the gutter
  is what disambiguates "who is talking" without reserving screen real
  estate.
- **WorkCards print only on turn completion** (done / interrupted /
  error). No in-place updating WorkCards. This is the explicit non-goal
  for v0.2 — see §H.
- **Status region** at the bottom of the screen widens from
  `⠁ @security working…` to `⠁ 3 roles working · @security · @ci · @backend`.
  Spinner advances from any role's event arrival, not on a wall-clock
  ticker only.

This keeps ssh-friendliness and the "no hidden state" property of v0.1
output — every WorkCard step the renderer ever drew is also in the
event log.

### F. Adapter cancel paths

Each adapter implements `cancel_turn(turn_id) -> Result<()>`:

- **cc** (`adapter/cc.rs`): write a `{"type":"interrupt"}` line on the
  long-lived stream-json stdin. cc handles graceful tool-call abort.
  If the in-flight turn does not match `turn_id`, no-op.
- **codex** (`adapter/codex.rs`): send JSON-RPC
  `notifications/cancelled { requestId }` on the MCP stdin. codex aborts
  the model loop and returns a partial result. The adapter folds the
  partial into a `RoleSpoke { turn_id, text: "[interrupted: <partial>]" }`.
- **gemini** (`adapter/gemini.rs`): no in-session interrupt. The per-turn
  `gemini -p` child is killed via SIGINT to its process group. The
  adapter then re-spawns on the next dispatch (gemini is per-turn already).

Each adapter sends `RoleStopped`-style closure events scoped to the turn
(not the role): `WorkCard { status: Interrupted, … }` finalization is
the renderer's job, not the adapter's.

## New / changed REPL commands

| Command | Behavior |
| --- | --- |
| `/stop` | Interrupt every in-flight turn. Roles stay alive. |
| `/stop @role` | Interrupt that role's in-flight turn (no-op if nothing in flight). |
| `/kill @role` | Terminate the role process. Equivalent to today's `/stop @role`. **Naming change** — old `/stop` semantics promoted to `/kill`. |
| `/refresh @role` | Unchanged. Still kills + respawns with composed priors. |
| Ctrl-C (1×) | Like bare `/stop`. REPL stays. |
| Ctrl-C (2× within 2s) | Exit REPL. Today's single-press behavior. |

`/help` and the splash tips need updating to reflect both the new
`/stop` semantics and the rename of process-kill to `/kill`.

## CREP changes (concrete delta)

```text
WorkTitle         { role, title }                   → +turn_id
RoleSpoke         { role, text, mentions, cost, … } → +turn_id
ToolCallProposed  { role, tool_name, …, tool_use_id }→ +turn_id
ToolCallExecuted  { role, tool_use_id, ok, … }      → +turn_id
PermissionDenied  { role, tool_name, … }            → +turn_id

(new)
TurnInterrupted   { role, turn_id, source }
                   source ∈ { user_stop, user_ctrl_c, watchdog_idle }
```

`TurnInterrupted` is a discrete event so renderers and `cr show` can
tell a clean stop from a model-said-it-was-done finish.

## Migration: four PRs

Each PR is independent-shippable. No single PR breaks v0.1 behavior.

### PR a — Plumbing

CREP gains `turn_id` (defaults to a synthetic single-turn for any caller
that doesn't supply one yet). `RoleHandle` gains `interrupt_tx`. All
three adapters implement `cancel_turn` no-ops or trivial paths. Tests
lock the field's serde shape and the adapter cancel signatures.

No user-visible behavior change. This PR is the foundation.

### PR b — Drain off the main loop

REPL refactor: drain task per role, dispatch separated from drain,
renderer subscribes to the bus directly. Per-turn WorkCards keyed by
`turn_id`. Status region widens. **Single-role behavior is unchanged**;
the REPL still feels serial because nothing yet dispatches more than
one turn at a time.

After this PR, parallel turns are *possible* but not yet *triggered* —
auto-router still goes one hop at a time, and the user has no way to
fire two prompts back-to-back because input still queues.

### PR c — Trust + interrupt UX

Delete `PER_TURN_TIMEOUT` and its branches. Implement `/stop`,
`/stop @role`, rename old `/stop` → `/kill`. Wire Ctrl-C two-press
semantics. Wire `interrupt_tx` end-to-end through each adapter's
`cancel_turn`.

Update `/help`, splash, README. CHANGELOG entry under `[Unreleased]`.

After this PR, the user can finally interrupt without exiting.

### PR d — Parallel dispatch

REPL accepts new prompts while turns are in flight. Auto-router becomes
fan-out aware: when role A's reply mentions @B, B's turn is dispatched
without waiting for anything else; A can also be re-dispatched
concurrently if the user @s it again. Per-role internal queue is
respected (one in-flight + N queued per role).

Status region grows to "N roles working" form. Renderer must handle
event-stream interleaving (role gutter is the only disambiguator).

After this PR, the v0.2 trinity is complete.

## Open questions (do not block PR a)

- **Auto-router fan-out cap.** Today's "one hop only" failure-mode
  mitigation is from v0.1. With parallel turns, a hop limit is still
  desirable (otherwise A→B→C→A loops are easy). Re-evaluate during PR d.
- **`/stop` echo policy.** When `/stop @role` interrupts mid-tool-call,
  do we echo the partial reply or just the WorkCard? Lean: echo
  partial reply, marked clearly. Decide when wiring PR c.
- **Auto-router and user-fired turns racing the same role.** Per-role
  serial means the second one queues. Does the renderer make the queue
  visible? Probably yes — a "@role queued · 1 ahead" status row.

## Out of scope for v0.2

These are explicitly **not** in this milestone:

- **In-place multi-WorkCard rendering.** Append-only is the chosen UX.
  Multi-card live updating is a TUI shape, conflicts with ssh aesthetic
  feedback.
- **`Esc` / single-key hotkey for interrupt.** Decided in design talk:
  pure command-style (`/stop`) wins over raw-mode hotkey. Reaction time
  cost (1–2s of typing) is accepted.
- **Parallel cap (`parallel_max` config).** Decided: trust users; no
  artificial ceiling. If a low-spec machine OOMs on eight engines,
  that's a user-tuneable config we can add later.
- **Cross-engine cache strategy.** Already deferred from v0.1; still
  deferred. Tracked in `proposed-amendments.md`.
- **Per-role human owners (team mode).** v0.x roadmap, not v0.2.

## Acceptance: what "done" looks like

A user opens `cr`, dispatches `@security` a 20-minute scan and
`@ci` a release-readiness check at the same time. Both run. The user
sees both event streams interleave with role gutters. At minute 4 the
user spots @security probing the wrong subsystem and types
`/stop @security`; @security's WorkCard finalizes as Interrupted, the
@ci card keeps streaming. The user dispatches `@security <new prompt>`
and that role picks up cleanly with refreshed direction. No turn was
killed by a wall clock. No one had to exit the REPL to get a word in.

That's the v0.2 promise.
