# CodeRoom v0.2 — Trust, Interrupt, Parallel

Status: design doc, **anchor for v0.2 PRs**. Lock spec here before code lands.
Anything that contradicts this doc goes to `docs/proposed-amendments.md` first.

This doc **amends** `docs/architecture.md` — it is not a soft companion. The
explicit amendment list is in §K. v0.1 constitution clauses that conflict
with v0.2 are flagged below and must be edited in lockstep with the
implementation PRs.

Origin: grew out of the May 2026 codex-event-routing fix (PR #80) and the
design conversation that followed. Reviewed by two independent agents (one
architecture-leaning, one adversarial); their findings shaped §A–§E and §H.

## TL;DR

v0.2 commits to three coupled changes that share one prerequisite: drain
the role turn out of the REPL's main control flow.

1. **Trust** — delete `PER_TURN_TIMEOUT`. The wrapper does not adjudicate
   when a model has "taken too long" by wall clock. Modern Claude / Codex
   / Gemini models self-terminate; the wrapper does not override that.
   The adapter keeps a **stdio protocol watchdog** (concrete numbers in
   §B) — a separate concept that triggers only on engine-stdio silence,
   never on slow-but-active model work.
2. **Interrupt** — every running turn is cancellable from the user prompt
   with `/halt @role` (single role), `/halt` (all in-flight), and `Ctrl-C`
   (all in-flight; second press exits REPL). `/stop` keeps its v0.1
   process-kill semantics. `/halt` is a new verb so v0.1 muscle memory is
   preserved.
3. **Parallel** — multiple roles can have active turns concurrently.
   Role-internally serial (one engine session, one turn at a time);
   role-externally parallel (no global drain).

These three are inseparable. Trust without interrupt = users locked into
runaway turns; trust without parallel = REPL prompt blocked by long
turns; parallel without interrupt = no escape hatch when a role goes off
the rails.

## §A. Why now (and why all three)

The v0.1.18 → PR #80 path made codex tool events visible in real time.
Once work is visible:

- The 5-minute `PER_TURN_TIMEOUT` becomes user-hostile. A real security
  scan or refactor takes 10–30 minutes. The user can see the work — the
  wrapper killing it on a wall clock is paternalistic and violates the
  Role Invariance Principle's promise that the user is the routing source.
- The user wants to react to what they see ("at minute 4 I noticed
  @security probing the wrong subsystem"). Without interrupt support,
  observation is decorative.
- Interrupting a single role is only useful if the REPL prompt stays
  reachable while that role runs. Today the prompt is blocked behind
  `drain_one_turn` (`src/repl.rs:577-634`).

The three changes share the same prerequisite (drain comes out of the
main loop) and the same justification (user-as-driver, not
wrapper-as-warden).

## §B. Trust: what it covers, what it doesn't

"Trust the model" applies to **model behavior**. It does **not** apply to
the IPC / RPC plumbing. Three layers, treated separately:

| Layer | Trusted? | v0.2 behavior |
| --- | --- | --- |
| Model self-termination | Yes | No timeout in REPL or adapter for "model took too long" |
| Engine stdio / RPC plumbing | No | Per-adapter idle watchdog, concrete numbers below |
| User control | n/a | `/halt`, Ctrl-C reachable while turn runs |

### Stdio watchdog: concrete numbers

The watchdog resets on every event flowing for the in-flight request.
Silence this long means the engine process or the bridge is wedged, not
that the model is thinking. On expiry the adapter emits
`TurnInterrupted { source: watchdog_idle }` and the role process is
left alive (the user's next dispatch reaches a working role).

| Adapter | Idle window | Rationale |
| --- | --- | --- |
| codex (`src/adapter/codex.rs`) | 10 min | PR #80 set 6 min; bumped because real codex turns can be quiet for several minutes mid-reasoning. |
| cc (`src/adapter/cc.rs`) | 15 min | cc emits stream-json events frequently; long silence is rare and indicates the subprocess is wedged. |
| gemini (`src/adapter/gemini.rs`) | 8 min | gemini is per-turn; wedged child = no exit. 8 min covers worst-case observed legitimate runs. |

These are configurable via `[engine_watchdog]` in `config.toml`. Default
on; setting `0` disables. Disable is supported but **off by default** —
silent wedges are a real failure mode and most users want the safety net.

This is a **distinction with bite**: the wrapper does not interrupt
based on time-since-dispatch; it interrupts based on time-since-last-
engine-event. A 30-minute scan emitting a tool call every 30 seconds is
never touched.

### Out of scope for the watchdog

Budget caps (`--max-budget-usd` on cc) are cost-bounded, not time-bounded;
they remain authoritative. `RoleStopped { reason: Budget }` retains its
v0.1 meaning. Trust covers timing decisions, not cost decisions — those
are user-set.

## §C. Architecture changes

### C.1 Turn IDs and thread IDs

CREP today carries only `role`. v0.2 adds two ids on every turn-scoped
event:

- **`turn_id`** — opaque per-turn id (ULID-shaped). Disambiguates events
  when multiple turns can be in flight against the same role (queued).
- **`thread_id`** — opaque per-conversation-thread id. Survives
  cross-role auto-routing: when @A in turn `T1` explicitly delegates to
  @B and the router dispatches B, B's turn carries the same `thread_id`
  as A's turn. Routing has no hop-depth counter; `thread_id` exists for
  ancestry, replay, and future parallel fan-out surfaces.

Why both: `turn_id` alone can't reconstruct the routing chain in a
parallel world (A→B and A→C are two different threads, not one), and
without `thread_id` replay cannot group the turns that belong to one
delegation chain.

### C.2 Interrupt signal channel

`RoleHandle` today exposes `stop_tx: oneshot<StopReason>`. One-shot, one
meaning: kill the role process. v0.2 splits the semantics:

- `stop_tx: oneshot<StopReason>` — unchanged. **Kills the role.** Used
  by `/refresh`, `/exit`, REPL teardown, `/stop` (which keeps v0.1
  semantics).
- `interrupt_tx: mpsc::Sender<TurnId>` — new. **Cancels the named turn,
  role process stays alive.** Used by `/halt`, `/halt @role`, Ctrl-C.

Adapter contract on receiving `InterruptTurn(turn_id)`:

- If `turn_id` matches the in-flight turn: cancel the engine request
  (per-adapter detail in §F). Cancel must complete within **5s** or the
  adapter escalates to `stop_tx` and emits
  `TurnInterrupted { source: cancel_timeout }`.
- If `turn_id` matches a queued-but-not-started turn: drop it from the
  queue, emit `TurnInterrupted { source: user_halt }`.

### C.3 Adapter session-kind taxonomy

Adapters are not isomorphic. v0.2 makes the distinction explicit via a
new capability:

```rust
enum SessionKind {
    /// One long-lived engine session. Queueing turns reuses session
    /// state (priors-cache, conversation memory). cc fits here.
    SessionBound,

    /// Each turn is a fresh engine invocation. Queueing turns is just
    /// scheduling; no inter-turn state. codex (v0.1) and gemini fit here.
    StatelessDispatch,
}
```

Implications for the queue:

- **SessionBound (cc)**: queueing preserves cached priors. A queued turn
  starts immediately when the previous turn ends.
- **StatelessDispatch (codex, gemini)**: each queued turn forks a fresh
  engine invocation. No state crossover. The "role process" is a
  dispatcher; `RoleStarted` is synthetic at spawn time.

This affects `/refresh` semantics too: refreshing a SessionBound role
matters (clears cached state); refreshing a StatelessDispatch role is
mostly a no-op (the next turn starts fresh anyway).

### C.4 REPL main loop

Goes from "dispatch + drain in one select!" to "dispatch only, render
off the bus":

```text
loop {
    select! {
        ctrl_c => interrupt_or_exit_two_press()
        line = stdin => parse_command_or_dispatch()
        evt = bus => render(evt); update_status_region()
        winch = signal::winch => repaint()
    }
}
```

The REPL is now **always interactive**. Even with eight turns in flight,
the prompt accepts new commands. No `tokio::time::timeout` wrapping any
turn drain.

### C.5 Per-role event demux (replaces shared broadcast subscribe)

v0.1 has a single `bus.subscribe()` consumer (`src/repl.rs:209`). With N
parallel drain tasks each calling `subscribe()`, all N consumers share
one 1024-slot ring (`src/bus.rs:27`); a slow consumer lags everyone.

v0.2 changes the topology:

- Bus stays broadcast for renderer + log writer (low-throughput consumers).
- A `RoleManager` task subscribes to bus, demultiplexes to per-role
  `mpsc<CrepEvent>(64)` channels.
- Each role's drain task owns its mpsc receiver. A wedged drain task
  only lags its own role.

This is mandatory before PR b (drain off main loop) ships. The shared
broadcast model does not survive parallel drains.

### C.6 Bounded queues + backpressure

`pending: VecDeque<DispatchedTurn>` in §C.4 has a per-role hard bound
of **8**. Reasoning: realistic auto-router fan-out + manual user
dispatch peaks under 8; unbounded queues are a self-DoS shape on long
sessions.

On dispatch when full:

```text
output::bad("@backend queue full (8 turns); /halt or wait");
```

Returns immediately. User decides whether to halt or wait.

## §D. CREP changes (concrete delta)

```text
WorkTitle         { role, title }                   → +turn_id, +thread_id
RoleSpoke         { role, text, mentions, cost, … } → +turn_id, +thread_id
ToolCallProposed  { role, tool_name, …, tool_use_id }→ +turn_id, +thread_id
ToolCallExecuted  { role, tool_use_id, ok, … }      → +turn_id, +thread_id
PermissionDenied  { role, tool_name, … }            → +turn_id, +thread_id
RoleStopped       { role, reason }                  → +turn_id: Option<String>
                                                      (Some when stopped during a turn)

(new)
TurnDispatched    { role, turn_id, thread_id, parent_turn_id: Option<String>,
                    queue_position: usize }
TurnInterrupted   { role, turn_id, thread_id, source,
                    partial_text: Option<String>,
                    partial_mentions: Vec<String> }
                  source ∈ { user_halt, user_ctrl_c, watchdog_idle, cancel_timeout }
```

Notes:

- `RoleStopped { reason: TimedOut }` is **retired** in v0.2. The wall
  clock is gone. Existing log readers must accept its absence.
- `RoleStarted` is unchanged; lifecycle, not turn-scoped.
- `parent_turn_id` on `TurnDispatched` carries auto-route ancestry.
  `thread_id` is shared across the chain; `parent_turn_id` reconstructs
  who dispatched whom (used by hop counting).
- `partial_text` is captured by the adapter from whatever the engine has
  emitted so far. **Mentions in a partial reply do not auto-route**
  (this is a deliberate safety choice — a half-finished thought
  shouldn't trigger downstream dispatches; the renderer surfaces them
  as a hint instead).
- `serde(deny_unknown_fields)` is **never** added to CREP types. v0.x
  log forward/backward compat depends on this.

## §E. New / changed REPL commands

| Command | Behavior | v0.1 → v0.2 delta |
| --- | --- | --- |
| `/halt` | Interrupt every in-flight turn. Roles stay alive. | **New verb.** |
| `/halt @role` | Interrupt that role's in-flight turn (no-op if nothing). | **New verb.** |
| `/stop @role` | Terminate the role process (kill). | Unchanged from v0.1. |
| `/refresh @role` | Kill + respawn with composed priors. | Unchanged. |
| Ctrl-C (1×) | Like bare `/halt`. REPL stays. | **Behavior changed**: was kill-all-and-exit. |
| Ctrl-C (2× within 2s) | Force `stop_tx` on every still-uninterrupted role, exit REPL. | Was first-press semantics. |

`/help`, splash, README, role priors templates need updating in PR b
(splash) and the README/architecture amendment list (PR c). Specific
v0.1 surfaces still saying `/stop = kill` (so they don't need text
edits — only `/halt` needs to be added to the lists):

- `README.md` (currently does not document `/stop`; needs `/halt` added)
- `src/repl/splash.rs:550` (help line)
- `docs/architecture.md:36, 362` (constitution — see §K)
- `docs/proposed-amendments.md:58, 82`

## §F. Adapter cancel paths (per-adapter)

Each adapter implements `cancel_turn(turn_id) -> Result<()>` with a 5s
upper bound. On expiry the wrapper escalates to `stop_tx` and re-spawns
on next dispatch.

### F.1 cc (`src/adapter/cc.rs`)

**Hypothesis**: stream-json stdin accepts a control message (likely
`{"type":"interrupt"}`) that aborts in-flight tool execution and emits
a `RoleSpoke` with whatever text was finalized.

**This hypothesis is unverified.** It is **not** present in the current
codebase. PR a ships a spike (`spike/L4-cc-interrupt.sh`) that probes a
real `claude --print --input-format=stream-json` to confirm. Outcomes:

- Spike succeeds: §F.1 keeps the stream-json interrupt path.
- Spike fails or message is rejected: cc cancellation falls back to
  `stop_tx + respawn`. Same UX (turn ends, role available next
  dispatch), higher cost (priors recomputed). This is the path
  `/refresh` already exercises (`src/repl.rs:768-789`); reuse it.

PR a does **not** ship cc cancellation; PR b/c picks the path based on
spike result.

### F.2 codex (`src/adapter/codex.rs`)

Send JSON-RPC `notifications/cancelled { requestId }` for the in-flight
`tools/call`. MCP 2024-11-05 specifies bidirectional cancellation; the
client→server direction (which is what we need) is documented but
codex-rs's specific implementation for `tools/call` mid-flight cancel
is unverified.

**Defensive default**: even if codex doesn't ack the cancel, the
adapter's `PendingRequestGuard` (`src/adapter/codex.rs:350-363`) drops
the orphan id, so the response future returns `RpcError::Closed` and
the in-flight `request()` resolves. The user sees `TurnInterrupted`
within the 5s SLO regardless of codex's cooperation. Background work
on codex's side may continue briefly; we ignore its eventual reply.

### F.3 gemini (`src/adapter/gemini.rs`)

Kill the per-turn child via SIGTERM → SIGKILL on the process group
(`terminate_child` at `:392-411`). v0.1 currently `abort()`s the stdout
reader (`:344`) and discards partial output. v0.2 changes this:

- Replace `tokio::spawn(read_pipe…)` with an mpsc-fed accumulator that
  feeds bytes into a per-turn `partial_text` buffer.
- On cancel, flush the accumulator into `TurnInterrupted.partial_text`
  before child termination.

Without this change, gemini interruptions produce empty WorkCards —
asymmetric UX vs cc/codex.

### F.4 cc adapter: WorkTitle dedup must be per-turn

`src/adapter/cc.rs:535-546` dedups `WorkTitle` events with a
per-process `bool` so the title only fires once. With pipelined turns
through one cc session, turn 2's title gets eaten because turn 1
already set the bool. v0.2 keys this on `turn_id` instead.

## §G. Render strategy: append-only, multi-role status

Reaffirms what the gutter work (commit `0afa871`) already enables.
v0.2 commits to it explicitly:

- **Per-event lines append in arrival order**, prefixed by the speaking
  role's color gutter. Multi-role event streams interleave; the gutter
  is the disambiguator. No reserved screen real estate per role.
- **WorkCards print append-only snapshots**: a first tool call may print
  a compact working card, and completion prints done / interrupted /
  error. No in-place updating WorkCards; that conflicts with ssh
  aesthetics.
- **Status region** is a **rewrite**, not a "widening". Today
  `StatusRegion` (`src/repl/status.rs:21-101`) is a single-slot
  spinner. v0.2 needs:
  - Global ticker owned by REPL select-loop (not per-turn).
  - `slots: Vec<StatusSlot>` populated/drained on `TurnDispatched`,
    drained on `TurnInterrupted` / `RoleSpoke`.
  - Width-aware ellipsis policy for "8 roles working · @a · @b · …".
- **Queued state visible**: `TurnDispatched.queue_position > 0` renders
  a "@role queued · 1 ahead" status row. No hidden state.

## §H. Edge case contracts

### H.1 Cancel SLO failure

If `cancel_turn(turn_id)` does not produce a turn-final event within
5s, the adapter:

1. Emits `TurnInterrupted { source: cancel_timeout, partial_text: <whatever was captured> }`.
2. Sends `stop_tx(StopReason::Crashed)` to its own wait_child task.
3. Role becomes unusable until the user `/refresh @role` or dispatches
   again (which auto-respawns).

### H.2 Ctrl-C #2 with cancels still pending

First press fans out `interrupt_tx` to every in-flight role. Some may
still be cancelling when the second press fires (within 2s). On second
press, the REPL **does not wait** for outstanding cancels — it sends
`stop_tx` to every role unconditionally and exits. Roles that hadn't
yet acknowledged cancel get the kill; their `TurnInterrupted` events
may arrive and be dropped (REPL is exiting anyway).

### H.3 Auto-route mention in a partial reply

When `/halt @security` interrupts mid-reply and `partial_text` contains
`@backend`, **the @backend mention does NOT trigger an auto-route**.
Half-thoughts shouldn't cascade. Renderer surfaces the partial mention
list as a hint:

```text
@security halted · partial reply mentioned: @backend (not dispatched)
```

User can manually `@backend <prompt>` if they want to follow up.

### H.4 Hop-depth in parallel

Hop-depth limit (architecture.md §"failure-mode mitigation") survives
v0.2 unchanged in spirit but moves to per-`thread_id` counting. A→B→C
is depth 3 on thread T; a separate user-fired A→D is depth 2 on a
different thread. The router rejects auto-dispatch that would push
`thread_id`'s hop count above the configured cap (default 3).

## §I. Migration: three PRs

Reduced from four after the b/c dependency was identified by review
(`drain_one_turn_with_timeout` cannot be split from `PER_TURN_TIMEOUT`
deletion).

### PR a — CREP shape + interrupt plumbing

CREP gains `turn_id`, `thread_id`, `TurnDispatched`, `TurnInterrupted`.
`RoleStopped` gains optional `turn_id`. `RoleHandle` gains
`interrupt_tx`. All three adapters implement `cancel_turn` (cc as a
no-op pending spike; codex with the real notification; gemini with the
killer + partial buffer).

`SessionKind` capability flag added next to existing `WorkTraceCapability`
(`src/adapter/mod.rs:74`).

PR a also ships:
- `spike/L4-cc-interrupt.sh` — probe stream-json interrupt support.
- Tests locking serde shape of every CREP event with the new fields.
- Per-role demux in `RoleManager` (no behavioral change yet — still
  one drain task globally, but the demux scaffolding is in place).

No user-visible behavior change. Foundation only.

### PR b — Drain off main loop + trust + interrupt UX (b+c merged)

REPL refactor: drain task per role, dispatch separated from drain,
renderer subscribes to bus directly, per-turn WorkCards keyed by
`turn_id`. Status region rewritten (per §G).

Delete `PER_TURN_TIMEOUT` and its `tokio::time::timeout` wrapper.
Implement `/halt`, `/halt @role`. Wire Ctrl-C two-press semantics
(first press = `/halt`, second = exit with force-stop_tx).

cc cancellation path chosen based on PR a's spike result. Idle
watchdogs (per §B) wired in adapters.

Update `/help`, splash, README, `docs/architecture.md` (per §K),
CHANGELOG `[Unreleased]`.

After this PR: trust is real, interrupt works, but the REPL still
single-tracks dispatch (auto-router and user input still serial).

### PR c — Parallel dispatch + auto-router fan-out

REPL accepts new prompts while turns are in flight. Auto-router
fan-out is parallel: when role A emits explicit delegation blocks for
@B and @C, both are dispatched concurrently with the same `thread_id`.
Per-role queue respected (one in-flight + ≤8 queued).

Grounding and explicit-delegation checks remain per turn; there is no
hop-depth counter.

After this PR: v0.2 trinity complete.

## §J. Out of scope

These are explicitly **not** in this milestone:

- **In-place multi-WorkCard rendering.** Append-only is chosen UX. TUI
  shapes conflict with ssh aesthetic.
- **`Esc` / single-key hotkey for interrupt.** Pure command-style
  (`/halt`) wins. 1–2s typing cost accepted.
- **`parallel_max` cap.** Per-role queue cap (8) is sufficient.
  Concurrent role count is unbounded — trust users.
- **codex-reply / multi-turn codex.** Tracked separately (`feat/adapter-codex-v2`).
- **Per-role human owners (team mode).** v0.x roadmap, not v0.2.

These are constitutional clauses; the amendment is structural, not

## §L. Acceptance: what "done" looks like

A user opens `cr`, dispatches `@security` a 20-minute scan and `@ci` a
release-readiness check at the same time. Both run. The user sees both
event streams interleave with role gutters. At minute 4 the user spots
@security probing the wrong subsystem and types `/halt @security`;
@security's WorkCard finalizes as Interrupted, the @ci card keeps
streaming. The user dispatches `@security <new prompt>` and that role
picks up cleanly with refreshed direction. No turn was killed by a
wall clock. No one had to exit the REPL to get a word in.

Edge cases that must also work in the acceptance test:

- `/halt @security` while codex MCP server is unresponsive → 5s SLO
  fires, role escalated to kill, next dispatch auto-respawns. WorkCard
  marked `cancel_timeout`.
- Ctrl-C while three roles are mid-cancel → second press within 2s
  forces stop on all, REPL exits cleanly, no zombie subprocesses.
- @host reply emits explicit @backend and @ci delegation blocks
  simultaneously → both dispatch in parallel with the same `thread_id`;
  prose mentions in summaries, tables, quotes, or code do not dispatch.

That's the v0.2 promise.
