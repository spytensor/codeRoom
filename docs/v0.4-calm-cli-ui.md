# v0.4.x Calm CLI UI

v0.4 treats the terminal surface as the product, not as a debug log.
CodeRoom still records the full CREP stream for audit, but the live REPL
only shows information that helps the user decide or act.

## Product rule

1. **Beautiful first.** The default screen should feel calm, aligned, and
   intentional. A CLI can be transparent without looking like stderr.
2. **Progress second.** While roles work, the user should know who is active,
   what they are doing now, whether anything needs approval, and how to halt.
3. **Information third.** Tool details, permission history, raw engine output,
   and adapter traces are audit data. They stay available through expansion,
   verbose mode, and `cr show`, not the default chat stream.

## Live visibility budget

| Event | Default live surface | Audit surface |
| ----- | -------------------- | ------------- |
| User input | Show | Transcript |
| Final role reply | Show | Transcript |
| Role started / work title | Hidden by default | CREP JSONL |
| Role stopped / interrupted | Quiet lifecycle line | CREP JSONL |
| Work title | Work-card title only | CREP JSONL |
| Tool proposed / executed | Summarize in WorkCard + status | Full `cr show`, `CODEROOM_VERBOSE_TOOLS=1` |
| Permission waiting | Show immediately; user action required | CREP / permission policy |
| Permission allowed | **Silent**; task continues | Policy file / engine transcript |
| Permission denied | Show; it changes task outcome | CREP JSONL |
| Auto-route handoff | One compact quote + handoff | CREP ancestry |
| Raw engine stderr | Hidden unless debug logging | `RUST_LOG` |

The key distinction: **waiting and denial are information; successful allow
is not.** An allow decision should clear the prompt row and disappear from the
live stream. Repeated auto-allowed calls must never print "allowed" lines.

## Work-card states

- **Working:** show role, title, spinner/current step, and at most a few recent
  steps. Do not stream every tool event as a separate chat line.
- **Waiting approval:** replace the status with a visible approval prompt for
  the relevant role/tool. The prompt is the interaction, not a log event.
- **Done:** collapse to one quiet summary line only when a persistent work card
  was shown or when the role produced no visible reply. Detailed steps stay in
  the trace.
- **Interrupted / denied:** remain visible because the expected flow changed.

## Message layout

- Role replies render as an inset message block: a single role header followed
  by indented body text.
- WorkCards are visually inset from the terminal edge. The leftmost column is
  not a global timeline.
- Auto-route quotes are one line. The parent message is already visible above,
  so repeated multi-line quotes are treated as noise.

## Non-goals for v0.4.x

- No full-screen ratatui rewrite.
- No GUI or hosted UI.
- No loss of auditability: every suppressed live event must still be available
  through the existing event log or engine transcript.
- No fake parity: engines that cannot expose a field still render `—` or omit it.

## Verification

Any PR touching this surface must include:

- Unit/snapshot coverage for WorkCard, permission prompt, and StatusRegion.
- Real terminal inspection at 60, 80, and 120 columns.
- Refreshed README images via `make readme-images`.
