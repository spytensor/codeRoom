# CodeRoom Feasibility Spike — 2026-05-09

Goal: before committing to an architecture, verify that the three target engines
(Claude Code, Codex CLI, Gemini CLI) expose enough common control surface to
build a multi-role wrapper without resorting to stdout/ANSI parsing.

## Control-surface comparison

| Capability                        | Claude Code 2.1                                  | Codex CLI 0.128                                  | Gemini CLI 0.32                                  |
| --------------------------------- | ------------------------------------------------ | ------------------------------------------------ | ------------------------------------------------ |
| Non-interactive driver            | `--print` / `-p`                                 | `codex exec` (subcmd) + stdin                    | `--prompt` / `-p`                                |
| Streaming JSON output             | `--output-format=stream-json`                    | `app-server` / `mcp-server` (RPC)                | `-o stream-json`                                 |
| Streaming JSON input              | `--input-format=stream-json` (over stdin)        | stdin via `codex exec -`                         | stdin appended to `-p`                           |
| **Hook events on stdout**         | `--include-hook-events`                          | none (use `mcp-server` mode)                     | `gemini hooks` subsystem (CC-compat)             |
| Structured output schema          | `--json-schema <schema>`                         | `--output-schema <file>`                         | (via response_schema in API)                     |
| System-prompt injection           | `--system-prompt`, `--append-system-prompt`      | `-c instructions=...` / config TOML              | extension/skill files + `--prompt`               |
| Dangerous-skip flag               | `--dangerously-skip-permissions`                 | `--dangerously-bypass-approvals-and-sandbox`     | `-y/--yolo` or `--approval-mode yolo`            |
| Granular approval modes           | `--permission-mode {auto,plan,…}`                | `-a {untrusted,on-request,never}`                | `--approval-mode {default,auto_edit,plan,yolo}`  |
| Session resume                    | `--resume <id>` / `--continue`                   | `codex resume` / `--last`                        | `--resume <id\|latest\|N>`                       |
| Cost cap                          | `--max-budget-usd <amount>`                      | (no built-in cap; rely on API limit)             | (no built-in cap; rely on API limit)             |
| MCP support                       | `--mcp-config`                                   | `codex mcp`, `codex mcp-server`                  | `gemini mcp`, MCP servers as extensions          |
| **Run as MCP server itself**      | no                                               | **yes** (`codex mcp-server`)                     | no                                               |
| Cross-engine protocol             | hooks (de-facto standard)                        | MCP                                              | `--experimental-acp` + hooks (CC-compat)         |
| Sandbox modes                     | (delegates to OS / claude-shipped sandbox)       | `-s {read-only,workspace-write,danger-full}`     | `-s` (basic) + `--policy`                        |

### Five things this table tells us

1. **All three speak `stream-json` (or equivalent structured output).** The wrapper's event source is structured, not terminal-scraped. The earlier "yellow" verdict was based on assuming we'd parse stdout; we won't.
2. **CC's `--include-hook-events` exposes hooks over stdout** when paired with `--print --output-format=stream-json`. We do not need filesystem hook configs.
3. **Gemini ships `gemini hooks migrate` to import CC hook configs.** Hook payload formats are compatible/portable. One hook spec covers two engines.
4. **Codex doesn't have a hook subsystem, but has `codex mcp-server`.** We talk to Codex over MCP (stdio JSON-RPC) instead — different shape, same outcome (structured event stream + structured tool-call interception).
5. **Gemini supports ACP (`--experimental-acp`)** — emerging cross-agent protocol. Bears watching but not a v0.1 dependency.

## What `claude --print --output-format=json` already gives us (verified)

A single one-shot run returned this envelope:

```json
{
  "type": "result", "subtype": "success",
  "session_id": "9970fecf-…",
  "total_cost_usd": 0.063,
  "modelUsage": { "claude-opus-4-7[1m]": {…}, "claude-haiku-4-5-…": {…} },
  "permission_denials": [],
  "terminal_reason": "completed",
  "iterations": [ … ],
  "usage": { "input_tokens", "output_tokens", "cache_*" }
}
```

This means even **without** `stream-json` mode, every CC call gives us:
- a stable session ID for routing
- a cost figure (per-call budget tracking is free)
- a `permission_denials` array (denials are first-class, not a side-effect of hooks)
- a structured termination reason
- per-model usage including cache hit/miss

That `permission_denials` field is the key: CC explicitly tracks denials as data,
not as silent failures. This makes the "permission gate may silent-fail" failure
mode much less likely than the earlier critique suggested — a denial leaves a
machine-readable trace.

## Re-verdict on each open question

### Q1. Permission semantics under skip-permissions
**Status: still needs empirical test, but risk lower than thought.**
- The `permission_denials` array exists in the result envelope. Denials are
  first-class data.
- `--max-budget-usd` provides a hard ceiling regardless of whether deny works
  on every individual call — a permission-gate bug cannot exfiltrate unbounded
  spend.
- Test plan: write a tiny `~/.claude/hooks/test-deny.sh` that rejects a
  specific Bash pattern, run `claude --dangerously-skip-permissions` against
  that pattern, observe `permission_denials`. **Manual test, user runs.**

### Q2. Live-session prompt injection
**Status: directly supported.**
- `--input-format=stream-json` over stdin is the documented mechanism.
- The wrapper writes new prompts as JSON messages on the persistent stdin
  pipe; CC continues the same session.
- No need to spawn a fresh subprocess per @-mention. Brief routing into a
  long-lived role session is the supported path.

### Q3. End-of-session journal writing
**Status: supported with `--json-schema`.**
- Provide a strict schema for the journal entry; CC validates output before
  returning. Reduces hallucination risk vs. free-form summary prompts.
- Codex has `--output-schema`; Gemini has API-level response schemas.
- Cross-engine: define a JournalEntry schema once, reuse across all three.

### Q4. Multi-engine adapter complexity
**Status: smaller than the positioning agent suggested.**
Three adapter strategies, ranked by simplicity:
- **(a) MCP-bridge mode for Codex**: run `codex mcp-server`, talk JSON-RPC.
  Cleanest. No CLI driving.
- **(b) `--print` driver for CC and Gemini**: spawn with `stream-json` I/O,
  read events as they arrive. Hook payloads are CC-compatible across both.
- **(c) Fallback for engines we add later**: text-mode with structured-output
  schema. Lossy but keeps adapter surface small.

A normalized internal event protocol (CodeRoom Event Protocol — CREP) would
have ~6 event types (RoleStarted, ToolCallProposed, ToolCallExecuted,
RoleSpoke, RoleStopped, PermissionDenied). Each engine adapter translates
native → CREP.

## Verdict

**GREEN. All three live tests passed.**

- Non-interactive structured-output drivers exist on all three engines.
- Hook/event capture works without stdout/ANSI parsing.
- Live-session prompt injection works (verified — see Live results below).
- Permission deny under `--dangerously-skip-permissions` works (verified).
- Codex MCP-server mode works (verified).
- Cross-engine adapter is reduced from "three custom hook subsystems" to "two
  stream-json adapters (CC, Gemini — hook payloads compatible) + one MCP
  client (Codex)".
- `permission_denials` and `--max-budget-usd` give us safety floors that don't
  rely on our hook code being bug-free.

## What this means for the architecture

Two changes from the previously-sketched architecture:

1. **The "Hook Adapter" layer becomes an "Engine Adapter" layer.** Each adapter
   speaks a normalized CREP. Engine-specific logic (CC's hook events, Codex's
   MCP messages, Gemini's hooks-import) is contained inside the adapter; the
   rest of the wrapper sees only CREP.

2. **Roles can pin engines.** A role's config is `{ priors_file, engine: "cc" |
   "codex" | "gemini", model: "..." }`. The Role Manager picks the right
   adapter at spawn time. This is the actual differentiator vs. existing
   tools — none lets you pin engine per role.

## Live results

Scripts in `spike/` (L1, L2, L3). Run separately because they need
a fresh `claude` subprocess and `--dangerously-skip-permissions` semantics
that don't make sense to nest inside a Claude Code agent session.

### L1 — permission deny under `--dangerously-skip-permissions` ✅

```
permission_denials length: 1
hook fired (input file exists)?: YES
first denial entry:
  { "tool_name": "Bash", "tool_input": { "command": "pwd", … } }
```

PreToolUse hook returning `{permissionDecision:"deny"}` blocks the Bash call.
Skip-permissions disables CC's *interactive* approval but preserves the hook
contract: our gate is sole arbiter when this flag is set, and denials are
recorded as structured data. **Permission gate design lives.**

Note (script v1 lesson): the prompt must be benign. v1's prompt mentioned
"DESPITE_DENY" / "PROOF_RAN_DESPITE_DENY" — those tokens triggered the model's
self-policing reflex, it refused all tool use, and the hook was never invoked.
v2 uses `Use the Bash tool to run pwd` and the hook fires normally.

### L2 — long-lived stream-json session ✅

```
result events:    2
unique sessions:  1
session_id:       2f9145cd-…
cache_read sequence: [17889, 26554]
```

Same session across multiple turns. `cache_read_input_tokens` jumped 17889 →
26554 — turn 1's working set (~8.6k tokens) became cache-hit on turn 2.
Long-lived role session + brief routing is feasible and economically sound.

Two operational notes from this run:

- **Pipe stdin via `cat $INPUT |`, not `< $INPUT`**. Bash file-redirect closes
  stdin too early; claude's stream-json reader sees EOF before processing.
  Pipe holds stdin open through the run. The wrapper will use a real pipe
  from a child process so this is not a concern in production code.
- **Pace writes by reading `result` events**. Three messages sent
  back-to-back produced only 2 result events (turns 2+3 merged) because the
  wrapper didn't wait between writes. This is documented SDK behavior — the
  wrapper must read until a `result` event arrives before sending the next
  user message.

### L3 — codex MCP-server handshake ✅

```
initialize protocolVersion: 2024-11-05
tools/list count:           2
tool names:
  codex
  codex-reply
```

Codex speaks MCP over stdio. The exposed tools are `codex` (start a session)
and `codex-reply` (continue a session). This is exactly the shape the wrapper
needs: structured RPC, no terminal scraping, native session management.

### L4 — gemini ACP exploration (deferred, informational)

Not run during this spike. Gemini's `--experimental-acp` flag is interesting
but Gemini's CC-hook-compatible mode (`gemini hooks migrate`) covers v0.1
needs. ACP is worth re-examining if/when we add a fourth engine.
